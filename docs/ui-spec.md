担当: @ささかまぼこ。

## 1. 目的

一人称視点の暗号推理ゲームに必要な画面レイアウト、UI、背景、キャラクター、モーション、素材制作の確定方針を定義する。

暗号文字は`mende-kikakui-font-guide.md`、ゲームルールは`game-rule.md`、実装責務は`implementation-spec.md`を正とする。

## 2. 全体方針

- 対象はPCブラウザ版とし、1920×1080の16:9を基準にブラウザ内で比率を維持して縮小する。
- Figmaの`最新仕様 2026-07-20`ページ（node `43:2`）を画面配置の正とする。
- Figmaは座標、寸法、照明、UI構成の正本とし、人物と机の最終画風は`public/assets/images`に配置した統一済みPNGを正本とする。
- プレイヤーは椅子に縛られた一人称視点で、移動操作はない。
- 中央奥に仮面の男、手前に机、机上に手帳とペン、通常会話は下部へ配置する。解答中と判定表示中は、透明な解答UIコンテナ内で問題文と解答欄だけを大枠へまとめ、候補、判定数、解答ボタン、操作案内をその外へ配置する。残り時間と間違い可能回数は右上へ表示する。
- 黒、こげ茶、暗い灰色、低彩度を中心に、暗く、不気味で、閉鎖的な雰囲気を優先する。
- 通常会話は白、暗号文は赤、日本語訳と判定前の解答は青を使う。
- 日本語UIは`Noto Sans JP Medium`、英字タイトルは`Inter Bold`、暗号文字だけは既存の`Noto Sans Mende Kikakui`を使う。
- 操作案内は会話・手帳などの本文領域外へ、小さな灰色で表示する。

### 背景と人物

- 背景は`#020202`とし、Figma node `13:66`から書き出した照明SVG、人物、机、ビネットだけで構成する。照明SVGは読込、開始、会話、問題、解答、判定、手帳、発砲、終了タイトル、リザルトの全場面で共通利用し、壁パネル、CSS製照明、別の照明素材、追加の背景グラデーションは置かない。
- 1920×1080では照明nodeを`x=612, y=-112.5, w=696, h=1020`とし、実際の照明画像をその枠内の左右7.4%、上1.78%、下25%のインセットへFigmaと同じ`object-fit: fill`で配置する。Figma SVGの形状と縦グラデーションは維持し、粒状フィルターは適用しない。通常人物はFigma基準位置から28px上げた`x=636, y=0, w=648, h=648`、机は`x=128, y=436, w=1665, h=797`相当へ配置する。
- 男はスーツ姿、正面向き、椅子に座った状態を基本とする。
- 照明はFigma node `13:66`のSVGで確定し、部屋の最終案だけをPM判断として残す。部屋はCSSで補完する。
- 人物差分は通常、銃を抜く、プレイヤーへ向ける、自分へ向ける4状態を使う。`masked-man-aim-self.png`を画風と服色の基準とし、他の3差分もほぼ黒のスーツ、暗いえんじ色、硬い多角形の面、低いハイライトへ統一する。差分切替時に服だけが明るく見えないよう、色調はPNGへ焼き込み、実行時のCSSフィルターでは補正しない。
- `desk-notebook-pen.png`の机、フクロウ、閉じた手帳、ペンも、人物と同じローポリゴン面、暗い木材色、抑えた金属光へ統一する。開いた手帳は`notebook-open-spread.png`を使う。

### 会話ボックス

- 1920×1080では`x=96, y=824, w=1728, h=144`相当のコンパクトな枠とする。
- 黒`rgba(3, 3, 3, 0.88)`、`rgba(150, 143, 125, 0.68)`の2px枠、6px角丸を使う。
- 男が喋るのは暗号例文、日本語訳、問題文だけとする。
- 発話主体は音声制御用の内部情報として保持するが、導入文、暗号例文、日本語訳、問題文のすべてで話者名と`「」`を表示しない。
- 導入文、暗号例文、日本語訳、問題単独提示の本文は、会話ボックス内で横方向の中央揃えにする。
- Figmaの進行表示と同じ役割の白いセリフ体表示を会話枠内の右下へ置き、導入・例文・翻訳の会話送りは`next→`、問題単独提示から解答受付へ入る時は`answer→`とする。`左クリックで進む`などのクリック操作案内は進行表示と会話枠から分離し、枠外の中央下へ置く。
- 暗号本文は赤、日本語訳は青、導入文と状況説明は白にする。

## 4. 暗号文字

- 暗号例文、問題文、選択対象トークン、手帳内暗号をすべて`Noto Sans Mende Kikakui`で表示する。
- 仮英字、Private Use Area、暗号画像は使用しない。
- Figmaとブラウザで同じTTF/OTF・WOFF2のソースとUnicode対応表を使う。
- Figma実行環境でローカルフォントを読み込めない場合は、同じWOFF2から生成した編集可能な輪郭を使い、レイヤー名へ元の実Unicodeを保持する。別フォントや別コードポイントへ置き換えない。
- 1単語はカテゴリ文字と候補文字の2文字で構成する。
- 問題内の単語順は左から右、単語内部は右から左にする。
- 2単語、3単語、Lv8の5単語で、境界と対応する日本語を追える余白を確保する。
- 通常会話、日本語訳、ボタン、リザルトへ暗号フォントを適用しない。
- 読み上げ用ラベルは`暗号単語1`のように順序だけを示し、意味を漏らさない。

## 5. 手帳

### 表示内容

- 手帳は通常画面上へ重ねるオーバーレイとし、背景を半透明の黒で暗くする。
- `notebook-open-spread.png`を見開き背景として使い、1920×1080では`x=100, y=60, w=1720, h=960`相当へ配置する。
- 提示済みの暗号例文と日本語訳に加え、正答した問題と解答を同じ表示形式で載せる。
- 履歴には発話者名と括弧を付けず、暗号文と日本語訳だけを表示する。
- `手帳`の見出しは表示しない。
- 左右同幅の見開きとし、時系列順に左ページへ3件、続いて右ページへ3件を置いてから次の見開きへ進む。
- 1920×1080では左ページ枠を`x=330`、右ページ枠を`x=974`、各620px相当とし、見開き画像の実際の紙面中央へ合わせる。履歴要素は560px相当とし、各ページ枠内で30px相当の左右余白を取る。
- 各履歴は紙色の半透明背景と枠を持たせ、暗号文と日本語訳の文字を従来より大きく、太くして紙面上で読み分けやすくする。背景の高さは内容に合わせ、翻訳の下に不要な空白を残さない。3件の配置段は維持する。5〜6件または5単語の履歴では収まる範囲で縮小する。
- ページ番号は`a/b`だけを最大24px相当で表示し、暗色の背景と枠を付けて見開き画像の上側外へ置く。縮小画面では画像と重ならない範囲まで文字と余白を縮小する。`Spaceで閉じる`と`A / Dでページを移動`はページ数に関係なく、同じ背景内の横並び操作案内として見開き画像の下側にある黒い余白へ置く。画像へ重ねず、狭幅時だけ2要素の組を保って折り返す。手帳内に閉じるボタンは置かない。

```text
┌────────── 2/3 ──────────────┐
│                                │
│ ┌────────────┬────────────┐ │
│ │ 暗号1・訳1  │ 暗号4・訳4  │ │
│ │ 暗号2・訳2  │ 暗号5・訳5  │ │
│ │ 暗号3・訳3  │ 暗号6・訳6  │ │
│ └────────────┴────────────┘ │
└──────────────────────────────┘
Spaceで閉じる    A / Dでページを移動
```

### 操作

- 問題文をクリックして解答受付へ入った後だけ、`Space`で開閉する。
- ゲームステージ内の本文、暗号、ラベル、ボタン文字は`user-select: none`とし、左クリックやドラッグによる範囲選択を表示しない。Spaceは手帳操作だけに使い、フォーカス中のボックスやボタンを選択・押下しない。
- 導入、例文、問題提示、判定中は手帳操作を受け付けない。
- 開いた直後は、最後に追加された履歴を含む見開きを表示する。
- `A` / `D`で前後の見開きへ移動し、先頭・末尾の範囲を越えない。
- 見開きが1つの場合も移動案内は表示し、A/Dの入力結果だけを現在ページの範囲内に留める。
- 5〜6件表示または5単語例文を含む場合はコンパクト表示とし、暗号文を約9〜13px、日本語訳を約9〜12pxへ縮小してスクロールなしで収める。
- `Tab`はゲーム操作に使わず、ブラウザ標準のフォーカス移動を維持する。

## 6. NEW通知

- 問題文をクリックして解答受付へ入り、現在レベルの例文を一括保存した瞬間に表示する。
- `answering`かつ手帳を閉じている時だけ、時間表示の12px下へ56×64px相当の手帳アイコンを表示する。アイコンは表示専用とし、クリック対象にしない。
- 未読時は手帳アイコンの上へ矢印なしの`NEW`を表示する。`answerFeedback`、手帳表示中、問題単独提示、会話、演出、リザルトではアイコンと通知を表示しない。
- 黒または暗い灰色を基本とし、くすんだ暗赤色を縁・影へ使用する。
- `NEW`文字だけを基準位置から上下4px動かす。1往復1800ms、`ease-in-out`、交互無限再生とする。
- `transform: translateY()`だけで動かし、レイアウト位置を連続変更しない。
- 通知自体はクリック対象にしない。
- 最初に手帳を開いた瞬間に既読となり、同じレベルでは再表示しない。
- 正答した問題と解答を追加した時は、NEW通知と書き留め音を発生させない。
- `prefers-reduced-motion: reduce`では静止表示にする。

## 7. 解答UI

### 基本操作

- 問題単独提示中は下部の通常会話ボックスだけを使い、残り時間と間違い可能回数を表示しない。
- 問題文をクリックした後の解答中と判定表示中は通常の会話ボックスを隠し、透明な解答UIコンテナを下部へ表示する。コンテナ内では、話者名と括弧を付けない問題文、暗号トークン、その直下の解答枠だけを半透明黒の問題・解答パネルへ収め、日本語候補、判定数、解答ボタン、操作案内はパネル外へ配置する。
- 問題文は赤いMende文字だけを表示する。
- 各暗号単語を、その直下に常設する枠付き解答欄との縦の組として並べる。
- 暗号単語を左クリックすると、選択中トークンの内部カテゴリに属する日本語候補2件だけを表示する。
- プレイヤー向け画面にカテゴリ名、品詞見出し、カテゴリ別色分けを出さない。
- 日本語候補を左クリックすると、選択した日本語を暗号単語の下へ表示する。
- 全トークンを回答するまで`解答する`を無効表示にする。
- 合否判定は`解答する`を押した時だけ行う。
- 1920×1080の16:9ではLv8の5組を可能な限り横一列に保ち、狭い画面では暗号単語と直下の解答枠を組のまま折り返す。高さが足りない時だけ透明な解答UIコンテナをスクロール可能にする。
- 1920×1080の解答時は透明な解答UIコンテナを`x=96, y=608, w=1728, h=440`とし、問題・解答パネルをその上部約190pxへ置く。トークン列はコンテナ基準で`x=106, y=22, w=1510, h=166`、候補列は`y=214`、解答ボタンは`y=296`、操作案内は`y=390`相当とする。
- 1920×1080の判定時は透明な解答UIコンテナを`x=96, y=590, w=1728, h=458`とし、`正答 n / N`を問題・解答パネルの上へ置く。候補列はコンテナ基準で`y=250`、解答ボタンは`y=326`、操作案内は`y=414`相当とする。

### 判定前

- 選択中トークンは青い枠または青い発光で示す。
- 選択済み日本語は青`#66aaff`、未選択は薄い横線または`未選択`とする。
- 日本語候補は共通の大枠へ入れず、各候補を150×54px相当の別々のボタンにする。各ボタンは`#111`背景、`#666`の2px枠、4px角丸、20px相当の青寄りの白文字とする。解答ボタンも暗い背景を維持する。
- ホバー、フォーカス、無効状態を色だけでなく枠や明度でも示す。

### 判定後

- 解答UI上部またはボタン付近に`正答 n / N`を表示する。
- 正答した日本語はくすんだ緑`#67c587`、誤答した日本語はくすんだ赤`#c75b5b`を文字と細い枠へ使う。可視の`正答`／`誤答`ラベルは付けない。
- 暗号文自体は赤のままにする。
- 全問正答の1400ms判定表示中は全操作を無効にする。終了条件となる2回目の誤答は判定表示を挟まずゲームオーバー演出へ直行する。
- 継続可能な1回目の誤答後は選択内容と色分けを残したまま操作とタイマーを即時再開する。同じ解答も再送信でき、日本語を変更した時は変更した解答欄だけ青い選択済み表示へ戻し、他欄の緑・赤と直前の`正答 n / N`を維持する。
- 継続可能な1回目の誤答時だけ問題・解答パネルを左右約10px、減衰付きで320ms揺らす。パネル外の候補、解答ボタン、判定数、操作案内は揺らさない。2回目の誤答では揺らさず、モーション低減時は1回目も揺らさない。
- Lv8の5単語でも、正答数と全トークンが同時に読めるよう折り返しを許可する。

## 8. 時間・間違い可能回数

- 残り時間は解答UIコンテナが出る`answering`と`answerFeedback`だけ右上へ白文字で表示する。導入、例文、問題単独提示中には表示しない。変更可能な既定値は90秒、残り15秒から赤にする。
- `間違い可能 n`も同じ表示条件で、時間表示の下へ小さく表示する。
- 初期値は1、0になったら赤で示す。
- 判定フィードバック中は解答UIコンテナと両ステータスを維持したままカウントを止める。手帳表示中はカウントを続け、開始・終了演出中はステータス自体を表示しない。
- 0になった瞬間に解答UIを終了し、判定表示なしでゲームオーバー発砲演出へ移る。

## 9. 開始演出

- 新規タブの初回起動とゲーム内リトライだけで再生する。同一タブのブラウザ再読込や開発時の再マウントでは、素材読込後に導入会話へ直接入り、完全暗転を再表示しない。
- まばたき用画像は制作しない。
- 現行版は画風統一済みの人物、机・手帳・ペン、開いた見開き背景、node `13:66`の照明SVG、CSS補完の部屋を表示し、実装側の黒い上下レイヤーでまぶたを描く。
- 通常時は次の2300msのタイムラインとする。

| 経過時間 | 視界 |
| --- | --- |
| 0〜300ms | 完全な暗転 |
| 300〜650ms | 細い隙間まで開く |
| 650〜820ms | 一度閉じる |
| 820〜1250ms | 半分程度まで開く |
| 1250〜1410ms | 短く閉じる |
| 1410〜2300ms | ゆっくり全開 |

- 演出中は入力を受け付けない。
- 配置済みの人物4差分、机画像、開いた見開き背景の読込失敗または5000msのタイムアウト時は、演出を始めずエラーと再読み込み案内を表示する。
- `prefers-reduced-motion: reduce`では、黒画面から300msで通常画面へフェードする。

## 10. 失敗・クリア演出

### 発砲まで

- 失敗時は、男が銃を抜き、プレイヤーへ向け、発砲して暗転する。
- クリア時は、男が銃を抜き、自分へ向け、発砲して暗転する。
- 紙芝居式の人物差分を使い、発砲フラッシュは1回、最大100msとする。
- 1920×1080では発砲時の人物差分も通常人物と同じ28pxだけFigma基準位置から上げ、`x=600, y=-48, w=720, h=720`相当、机を`x=128, y=436, w=1665, h=797`相当へ配置し、画面全体へビネットを重ねる。
- 発砲演出中もFigma node `13:66`の照明を通常画面と同じ`x=612, y=-112.5, w=696, h=1020`相当で表示する。発砲フラッシュと発砲直後の暗転だけは演出として一時的に照明を覆う。
- Figmaの状態名・タイミング注釈と、発砲内容を説明する字幕はプレイヤー画面へ描画しない。

### 終了タイトル

- 発砲・暗転後、失敗時は`GAME OVER`、クリア時は`GAME CLEAR`をテキストで表示する。画像素材は作らない。
- 終了タイトルの黒地にもFigma node `13:66`の照明を通常画面と同じ位置へ表示し、タイトルを照明より前面へ置く。
- タイトルは`Inter Bold`の138px相当とする。
- `GAME OVER`は`#b23b3b`で衝撃表示と小さな横揺れを行い、2300ms表示する。
- `GAME CLEAR`は`#e8e0c2`で、浮上、字間収束、淡い光を行い、2400ms表示する。
- モーション低減時は1500msの単純なフェードにする。
- タイトル完了前にリザルトを表示しない。

## 11. リザルト

- `#020202`の黒地にFigma node `13:66`の照明だけを置き、人物、机、手帳、ビネットは表示しない。
- 見出しは終了タイトルと重複させず`RESULT`とする。
- 経過時間、正解回数、失敗回数を白で表示し、数値を少し大きくする。
- 1920×1080では中央に900×600相当のパネルを置き、ラベルと値を2列で配置する。
- 画面下部に`左クリックでリトライ`案内を表示する。専用のリトライボタンは表示しない。

## 12. 音の表示タイミング

- 男の発話音は、暗号例文、日本語訳、問題文が表示された瞬間だけ鳴らす。
- 会話送り音は左クリックで会話を進めた時に鳴らす。
- 有効なボタンをマウスまたは`Enter`で押した時は、会話送り音を押下音として1回鳴らす。無効ボタン、`Space`、ボタン以外の背景クリックには追加しない。
- 書き留め音は解答受付へ入り、例文の一括記録とNEW表示を開始した時に鳴らす。
- 銃を抜く音と発砲音は対応する演出で鳴らす。
- `end.mp3`は終了タイトルの開始時に1回だけ鳴らす。
- `close-note.mp3`はSpaceで手帳を閉じた時に1回だけ鳴らす。
- `open-note.mp3`はSpaceで手帳を開いた時と、A/Dで見開きが実際に移動した時に1回だけ鳴らす。
- `wrong-answer.mp3`は解答送信で誤答が確定した時に1回だけ鳴らす。継続可能な1回目と即終了する2回目の両方を対象とし、時間切れ、正解、再描画では鳴らさない。
- 全音源は起動時に3要素ずつ先読みし、音声読込のためにゲーム開始を待たせない。
